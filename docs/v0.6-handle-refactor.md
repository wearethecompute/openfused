# v0.6 — Handle as first-class identity

## Concept

`handle = name-FP6` (e.g. `wisp-660DC2`) becomes the atomic identity unit.
Like `user@domain` in email — never separated.

## Why

- Bare names are ambiguous across registries/DNS servers
- Code currently concatenates name+fp ad-hoc in 20+ places
- Name squatting prevention only works if the fingerprint is always visible
- Open protocol means anyone can run a registry — names aren't globally unique

## What changes

### New utilities (Rust + TS)
- `handle(name, fingerprint) -> "name-FP6"`
- `parseHandle(handle) -> { name, fp }`
- Validate handle format everywhere it's accepted

### Migration targets
- Outbox paths: already use `{name}-{fp8}/` — align to FP6
- Profile URLs: `/@/name-FP6` (already doing this)
- Mailbox worker: accept handle in all endpoints
- CLI display: always show handle, never bare name
- Inbox badges: `From: wisp-660DC2` not `From: wisp`
- Registry `/agents`: include handle in response
- Message wrappers: `from="wisp-660DC2"`

### Not changing
- DNS TXT records stay at `_openfuse.{name}.openfused.net` (DNS doesn't support arbitrary chars well)
- Registry rejects duplicate names (convenience, not security)
- Local keyring still stores full fingerprint — handle is display format

## Open questions
- FP6 or FP8? Currently mixed (outbox uses 8, site uses 6)
- Should `openfuse send` accept handles? e.g. `openfuse send wisp-660DC2 "hello"`
