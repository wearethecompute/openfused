# WASM Demo — Browser-Based OpenFused Core

**Status**: OPEN
**Assignee**: _browser claude_
**Created**: 2026-03-28

## What You're Building

A browser-based demo that runs `openfuse-core.wasm` — proving OpenFused's "runs anywhere" story. The WASM binary already exists and works in Node.js via `node:wasi`. Your job is to make it work in a browser.

## Context

OpenFused is a file-native protocol for AI agent context. The core library (Rust) compiles to `wasm32-wasip1` and handles: key generation, Ed25519 signing/verification, age encryption/decryption, store initialization, inbox read/send, config management.

The WASM binary is at: `wasm/openfuse-core.wasm`
The Node.js wrapper is at: `src/wasm-core.ts`

The WASM module uses WASI (WebAssembly System Interface) — it expects a filesystem via preopens, reads args, and writes JSON to stdout. In Node.js this works via `node:wasi`. In the browser, you need a WASI polyfill.

## How The WASM Core Works

It's a CLI-style binary. You pass subcommands via args, it reads/writes files via WASI preopens, and returns JSON on stdout.

```
args: ["openfuse-core", "init", "demo-agent", "abc123"]
→ creates .mesh.json, .keys/, CONTEXT.md, PROFILE.md in the preopened dir
→ stdout: {"name":"demo-agent","id":"abc123","publicKey":"...","encryptionKey":"..."}

args: ["openfuse-core", "sign-message", "--from", "demo-agent", "--message", "hello world"]
→ reads .keys/private.key from preopened dir
→ stdout: {"from":"demo-agent","message":"hello world","signature":"...","publicKey":"...","timestamp":"..."}

args: ["openfuse-core", "read-config"]
→ reads .mesh.json from preopened dir
→ stdout: {"id":"...","name":"...","publicKey":"...","peers":[],"keyring":[]}

args: ["openfuse-core", "status"]
→ reads store state
→ stdout: {"name":"...","inbox":0,"shared":0,...}
```

See `src/wasm-core.ts` for the full list of subcommands and their args/returns. The `callWasm()` function shows exactly how Node.js invokes it.

## What The Demo Should Do

A single-page app (HTML + JS, no framework) that demonstrates:

1. **Init** — Click "Initialize Agent" → runs `init` via WASM → shows the generated keys and config
2. **Sign** — Type a message, click "Sign" → runs `sign-message` → shows the signed JSON envelope
3. **Verify** — Paste a signed message, click "Verify" → runs `verify-message` → shows valid/invalid
4. **Encrypt** — Type a message + recipient age key, click "Encrypt" → runs `sign-and-encrypt` → shows encrypted envelope
5. **Status** — Click "Status" → runs `status` → shows store state

Each step should show the raw JSON output and the files that were created/modified in the virtual filesystem.

## Technical Approach

### WASI in the browser

Use `@aspect-build/aspect-wasi` or `@aspect-build/aspect-wasi-js` — these are browser-compatible WASI polyfills that provide an in-memory filesystem. Alternatively, use `browser_wasi_shim` (npm: `@aspect-build/aspect-wasi`).

Actually, the most maintained option as of early 2026 is **`@aspect-build/aspect-wasi`** or **`browser_wasi_shim`** from the bytecodealliance. Look up what's current — the WASI browser polyfill landscape changes fast. Key requirement: it must support WASI preview1 (`wasi_snapshot_preview1`) with preopens and stdout capture.

The pattern:
```js
// 1. Load the WASM binary
const wasmBytes = await fetch('/wasm/openfuse-core.wasm').then(r => r.arrayBuffer());

// 2. Set up WASI with in-memory filesystem
const wasi = new WASI({
  args: ['openfuse-core', 'init', 'demo-agent', 'demo123'],
  preopens: { '/store': memoryFs },  // in-memory FS mapped to /store
});

// 3. Instantiate and run
const module = await WebAssembly.compile(wasmBytes);
const instance = await WebAssembly.instantiate(module, wasi.getImportObject());
wasi.start(instance);

// 4. Read stdout for JSON result
const output = wasi.getStdout(); // depends on polyfill API
```

### File structure

Put the demo in `site/demo/` (the `site/` directory is the landing page, deployed to openfused.dev):

```
site/demo/
├── index.html      ← the demo page

└── style.css       ← minimal styling
```

The WASM binary is served from `/wasm/openfuse-core.wasm` (or bundled).

### Important details

- The WASM binary is ~900KB (already optimized with wasm-opt)
- It needs WASI preview1 — NOT preview2, NOT wasm32-unknown-unknown
- The preopened directory must be `/store` (that's what the Rust code expects as its working directory)
- Stdout capture is how you get results — the WASM binary prints JSON to stdout
- The in-memory filesystem persists across calls within the same page session (so init creates keys, then sign-message can read them)
- Each WASM call needs a fresh instance (WASI modules are run-once via `_start()`)

### Don't

- Don't use React/Vue/Svelte — plain HTML + vanilla JS
- Don't use a bundler — ES modules loaded directly
- Don't over-design the UI — this is a technical demo, not a product
- Don't try to implement networking (sync, registry) — WASM core is crypto + store only

### Do

- Make it look clean and minimal
- Show the raw JSON for every operation (developers will want to see the wire format)
- Show the virtual filesystem state (what files exist, their contents)
- Include a "copy to clipboard" on signed messages so people can verify each other's signatures
- Add a brief explanation of what OpenFused is at the top
- Link to https://openfused.dev and https://github.com/openfused/openfused
